# Supabase — Phase 1 (schema + data migration)

This directory holds the Postgres schema and seed for the B2C auth migration.
**Phase 1 is schema + data only** — no API endpoint is gated and no login screen
exists yet. The existing `data/positions/*.json` and `data/portfolio/portfolio.json`
remain the **live source** until Phase 2 flips the API over to Postgres. Standing
this up changes **no current behaviour**.

## Migrations

| File | What it does |
|---|---|
| `migrations/0001_init.sql` | Creates `profiles`, `positions`, `portfolio_holdings` with RLS (`user_id = auth.uid()`). |
| `migrations/0002_seed_official_beta.sql` | Seeds the pre-auth, single-tenant beta history as **official-beta** rows (`user_id IS NULL`, `is_official_beta = true`). Generated by `scripts/migrate_positions_to_supabase.py`. |

## Design (from Phase 0 investigation)

- **Global "official beta" record (Q2):** the current single-tenant positions are
  preserved as `positions` rows with `user_id IS NULL` + `is_official_beta = true`.
  A `CHECK (user_id is not null or is_official_beta)` guarantees every row is
  either a real user's row or an official-beta row. Regular users get their own
  rows (`user_id = auth.uid()`), which never feed the official beta dashboard.
- **RLS:** users can see/manage only rows where `user_id = auth.uid()`. Official-beta
  rows (NULL `user_id`) match no user, so they are invisible via user RLS — the
  global beta dashboard reads them via the service role / a dedicated view (Phase 2).
- Daily cron (Q1) and per-user portfolio alerts (Q3) are Phase-2 wiring; no schema
  impact here.

## Applying (manual — requires your Supabase project)

No Supabase credentials exist in CI/this repo, so the SQL is authored but not
applied here. To apply against your project:

```bash
# Supabase CLI (recommended)
supabase link --project-ref <your-project-ref>
supabase db push        # runs migrations/*.sql in order

# …or paste 0001_init.sql then 0002_seed_official_beta.sql into the SQL editor.
```

Regenerate the seed from the current JSON at any time (deterministic, idempotent to review):

```bash
python scripts/migrate_positions_to_supabase.py          # generate + verify
python scripts/migrate_positions_to_supabase.py --check   # verify fidelity only
```

## Environment variables (added in Phase 1)

`SUPABASE_URL`, `SUPABASE_ANON_KEY` (public/frontend-safe) and `SUPABASE_JWT_SECRET`
(**backend only**). See `.env.example` and `product/config.py`. Existing vars
(`NEWS_API_KEY`, `ALLOWED_ORIGINS`, `PORT`, `EDGAR_USER_AGENT`) are untouched.

## Verifying the official beta

After `db push`:

```sql
select ticker, status, entry_date, exit_date, realized_return, days_held
from public.positions
where is_official_beta;
-- expect 0 rows. The official beta record starts CLEAN — the only prior row
-- (ORCL, entry==exit 2026-06-12) was confirmed demo data from MVP development
-- (commit 180254a), not a real trade, and was removed. Real official-beta rows
-- accrue only when the live automation opens/closes genuine positions.
```
